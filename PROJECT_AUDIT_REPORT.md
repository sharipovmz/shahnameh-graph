# PROJECT_AUDIT_REPORT

## 1. Общая структура проекта

Проект представляет собой Streamlit-приложение для визуализации и анализа графа знаний по мотивам `Шоҳнома`. На практике это не крупная многослойная система, а компактное аналитическое веб-приложение с разделением на:

- точку входа приложения;
- модуль загрузки и нормализации данных;
- модуль построения графа и вычисления метрик;
- модуль UI;
- модуль визуализации PyVis;
- дипломный текст в markdown/docx;
- CSV-датасет как основной источник данных.

Корневая структура проекта:

- `app.py` — основной Streamlit entrypoint.
- `main.py` — старый автономный скрипт генерации HTML-графа.
- `data_tajik.csv` — исходный датасет связей.
- `shahnameh_app/` — пакет с прикладной логикой.
- `diploma_thesis.md` — текст дипломной работы.
- `diploma_thesis.docx` — DOCX-версия диплома.
- `549acda423eda79a30443d69387f3777_1605_27467.docx` — сторонний DOCX-образец/методический источник.
- `tajik_graph.html` — статический экспорт графа.
- `venv/` — локальное окружение.

Итог: архитектурно проект ближе к хорошо структурированному прототипу/демонстрационному приложению, чем к промышленной системе.

## 2. Назначение ключевых файлов и папок

### `app.py`

Главный файл запуска. Выполняет:

- конфигурацию страницы Streamlit;
- загрузку данных;
- построение метаданных узлов;
- получение пользовательских фильтров из sidebar;
- фильтрацию рёбер;
- построение графа NetworkX;
- расчёт метрик;
- рендер интерфейса и встраивание PyVis HTML.

Это orchestration-layer, без тяжёлой логики внутри.

### `shahnameh_app/constants.py`

Набор базовых констант:

- обязательные колонки CSV;
- фильтры по типам и сторонам;
- параметры высоты графа.

Файл маленький, но полезный для устранения магических строк.

### `shahnameh_app/data.py`

Отвечает за:

- чтение CSV;
- базовую валидацию структуры;
- очистку пробелов;
- построение словаря метаданных узлов;
- эвристическое определение атрибутов узлов, которые встречаются только как `Target`;
- фильтрацию рёбер по выбранным условиям.

Это один из ключевых модулей проекта, потому что в нём сосредоточены допущения модели данных.

### `shahnameh_app/graph_logic.py`

Отвечает за:

- сборку `networkx.DiGraph`;
- суммарные метрики;
- degree centrality;
- betweenness centrality;
- подготовку таблиц для UI.

Файл компактный и логически чистый.

### `shahnameh_app/ui.py`

Самый прикладной модуль UI. В нём:

- sidebar;
- hero/header;
- секционные интро-блоки;
- карточки метрик;
- вкладки с таблицами;
- легенда;
- expander с данными;
- блок помощи;
- footer.

Это основной слой представления.

### `shahnameh_app/styles.py`

Крупный CSS-модуль, задающий:

- две темы: `dark` и `light`;
- дизайн-систему через CSS variables;
- кастомный стиль Streamlit-компонентов;
- hero-блок, панели, табы, expander, таблицы, sidebar и визуальные токены.

Файл большой для проекта такого масштаба и является центром визуального поведения.

### `shahnameh_app/visualization.py`

Генерирует HTML-граф через PyVis:

- тематические настройки графа;
- цвет и форма узлов;
- добавление узлов и рёбер;
- временный HTML-файл;
- возврат HTML-строки в Streamlit.

Это bridge между аналитической моделью и интерактивной визуализацией.

### `main.py`

Старый альтернативный скрипт, который:

- напрямую читает CSV;
- строит PyVis-граф;
- сохраняет `tajik_graph.html`.

Сейчас выглядит как legacy/debug utility, а не часть основного приложения.

### `tajik_graph.html`

Статический экспорт старого графа, вероятно результат запуска `main.py`.

### `diploma_thesis.md`

Ключевой не-кодовый файл проекта:

- описывает академическую рамку;
- содержит план диплома;
- фиксирует позиционирование проекта;
- показывает расхождение между заявленным масштабом и текущей реализацией.

## 3. Анализ UI/интерфейса

### Общая оценка

Интерфейс уже не выглядит как голый Streamlit. Видно явное стремление сделать продукт презентабельным:

- hero-блок;
- кастомные панели;
- цветовые темы;
- метрики;
- легенда;
- таблицы;
- поясняющие блоки.

Это сильная сторона, особенно для дипломной демонстрации.

### Что сделано хорошо

- Есть единая структура страницы: вводный блок → метрики → аналитика → граф → данные → помощь.
- Есть sidebar с управлением.
- Есть переключатель тем.
- Есть визуальная легенда графа.
- Есть таблицы и экспорт CSV.
- Есть пояснение метрик, что снижает порог входа для комиссии.

### UX-плюсы

- Интерфейс понятен даже без чтения кода.
- Пользователь быстро понимает, что можно фильтровать данные.
- Есть разделение аналитики и исходных данных.
- Приложение пригодно для демонстрации на защите.

### UX-минусы

- Много смысловой нагрузки сосредоточено в одном экране.
- Граф может стать трудно читаемым при росте датасета.
- Табличный интерфейс Streamlit остаётся нестабильной зоной из-за кастомизации.
- Отсутствует отдельный onboarding или сценарий “что именно смотреть сначала”.

### Риски UI

- Тяжёлый CSS для Streamlit может ломаться при обновлении Streamlit.
- Кастомизация `st.dataframe` уже показала нестабильность.
- Светлая тема и внутренние таблицы требуют осторожной поддержки.

## 4. Анализ backend / логики

### Общая оценка

Backend в привычном серверном смысле отсутствует. Это локальная аналитическая логика внутри Streamlit-приложения.

Архитектурно логика простая:

1. загрузить CSV;
2. нормализовать данные;
3. построить метаданные;
4. применить фильтры;
5. построить `DiGraph`;
6. вычислить метрики;
7. визуализировать результат.

### Сильные стороны логики

- Хорошее разбиение по модулям.
- Нет чрезмерной связанности между слоями.
- Код легко читать.
- Функции имеют ясную ответственность.

### Слабые стороны логики

- Логика данных опирается на эвристику для `Target`.
- Отсутствует слой явной схемы/онтологии.
- Нет тестов.
- Нет устойчивой обработки конфликтных/грязных данных.
- Нет кэширования данных и вычислений Streamlit-средствами.

### Технические риски

- `filter_edges` делает построчный `apply`, что плохо масштабируется.
- При росте корпуса производительность может резко упасть.
- Семантика узлов зависит от того, как они появились в CSV.
- Метрики считаются на текущем срезе, а не на полной нормализованной модели.

## 5. Анализ работы с данными

### Что есть сейчас

Датасет очень маленький: `56` строк.

Структура:

- `Source`
- `Relationship`
- `Target`
- `Type`
- `Side`

Это минимально достаточный формат для демонстрации графа.

### Сильные стороны

- Простая и прозрачная таблица.
- Удобна для ручного редактирования.
- Хорошо подходит для учебного проекта и первой итерации.

### Слабые стороны

- Нет уникальных идентификаторов сущностей.
- Нет нормализованной схемы.
- Нет ссылок на главы/эпизоды текста.
- Нет событий как самостоятельных сущностей.
- Нет временного измерения.
- Нет механизма валидации против справочника имён.

### Качество данных

В CSV видны проблемы качества:

- сомнительные формы: `Љарира`, `Рўҳҳом`;
- возможная опечатка: `Паҳлавол`;
- смешение сущностей разных типов;
- `Side` для ряда объектов неочевиден;
- местами сущности отражают персонажей, местами группы, местами артефакты, местами географию.

Это главный риск для диплома: слабое качество данных подрывает доверие к аналитике.

## 6. Архитектура приложения

### Архитектурный стиль

Приложение реализовано в стиле “тонкий entrypoint + специализированные модули”.

Слои:

- `app.py` — orchestration;
- `data.py` — ingestion and normalization;
- `graph_logic.py` — analytics;
- `visualization.py` — rendering bridge;
- `ui.py` — presentation;
- `styles.py` — theme/design layer.

### Почему это хорошо

- Легко ориентироваться.
- Проект удобно объяснять на защите.
- Простая расширяемость.

### Почему это пока не зрелая архитектура

- Нет схемы доменной модели.
- Нет тестового слоя.
- Нет persistence layer кроме CSV.
- Нет API-слоя.
- Нет разделения между research model и presentation model.

## 7. Разбивка по важным файлам и модулям

### `app.py`

Роль:
- координирует весь runtime.

Плюсы:
- чистый и компактный;
- легко читать;
- минимально перегружен.

Минусы:
- все действия выполняются в одном линейном потоке;
- нет кэширования;
- нет логирования.

### `shahnameh_app/data.py`

Роль:
- загрузка, валидация, фильтрация, fallback-атрибуты.

Плюсы:
- есть валидация обязательных колонок;
- есть очистка строк.

Минусы:
- fallback-логика слишком грубая для гуманитарного проекта;
- нет нормализации имён;
- нет устранения дублей.

### `shahnameh_app/graph_logic.py`

Роль:
- построение графа и метрик.

Плюсы:
- аккуратная изоляция аналитики;
- понятные функции;
- удобно использовать в дипломном описании.

Минусы:
- аналитика пока поверхностна;
- только две метрики;
- нет слоя интерпретации результатов.

### `shahnameh_app/ui.py`

Роль:
- рендер всех видимых секций интерфейса.

Плюсы:
- хорошая структурность;
- много переиспользуемых функций;
- интерфейс ориентирован на демонстрацию.

Минусы:
- часть формулировок всё ещё требуют академической доработки;
- сильная завязка на HTML/CSS внутри Streamlit;
- возможна хрупкость при доработках.

### `shahnameh_app/styles.py`

Роль:
- весь визуальный язык приложения.

Плюсы:
- темы;
- сильная визуальная идентичность;
- хорошая демонстрационная ценность.

Минусы:
- файл слишком велик для такого проекта;
- много низкоуровневой кастомизации;
- высокая вероятность будущих регрессий UI.

### `shahnameh_app/visualization.py`

Роль:
- конструирование графа PyVis.

Плюсы:
- простая и понятная логика;
- согласована с темами;
- подходит для наглядной защиты.

Минусы:
- работает через временный HTML-файл;
- PyVis слабее контролируется, чем полностью кастомный frontend;
- визуализация ограничена по масштабированию и удобству анализа.

### `main.py`

Роль:
- legacy-скрипт быстрой генерации HTML.

Проблема:
- создаёт дублирование логики;
- может путать при чтении проекта;
- лучше воспринимать его как архивный артефакт.

### `diploma_thesis.md`

Роль:
- академическое позиционирование проекта.

Плюсы:
- задаёт структуру диплома;
- показывает, как проект должен быть защищён в текстовом виде;
- полезен для связи кода и дипломной логики.

Минусы:
- в некоторых местах заявляет больший масштаб и методологическую глубину, чем есть в коде и данных.

- Есть работающий результат, а не только описание.

- Интерфейс уже пригоден для презентации.

## 9. Слабые стороны проекта

- Очень маленький датасет.
- Слабая онтология.
- Нет формального слоя валидации данных.
- Нет тестов.
- Нет устойчивой стратегии масштабирования.

## 10. Риски и технический долг

### Технический долг

- `main.py` и `tajik_graph.html` как устаревшая ветка подхода.
- крупный CSS-файл со сложной кастомизацией Streamlit.
- отсутствие automated tests;
- отсутствие typed domain model;
- отсутствие нормализованного data dictionary.

### Риски

- поломка UI при обновлении Streamlit;
- семантические ошибки при расширении датасета;
- некорректные выводы из графовых метрик;
- критика комиссии из-за слабой репрезентативности корпуса;
- путаница между “исследовательским инструментом” и “научным доказательством”.

## 11. Что важно для дипломной работы

Для диплома этот проект важен не только как программа, но и как академический кейс.

Что можно уверенно защищать:

- цифровизация культурного наследия;
- проектирование модели данных;
- разработка информационно-аналитического инструмента;
- визуализация и анализ отношений;
- практическая применимость в образовании и цифровой гуманитаристике.

Что опасно защищать без оговорок:

- “полная модель Шоҳнома”;
- “объективное выявление главных героев”;
- “глубокий литературоведческий анализ” только на основе current graph;
- сильные выводы о художественной значимости персонажей.

Что нужно обязательно подчеркивать в дипломе:

- проект — это цифровая модель ограниченного корпуса;
- графовые метрики — вспомогательный аналитический инструмент;
- система предназначена для поддержки анализа, а не для замены филологической интерпретации;
- расширение корпуса и онтологии является следующим этапом работы.

## 12. Оценка проекта с точки зрения специальности «программист в экономике»

С позиции этой специальности проект выглядит лучше, чем с позиции чистого литературоведения.

Почему:

- есть информационная система с понятной архитектурой;
- есть работа с данными;
- есть визуализация;
- есть пользовательский интерфейс;
- есть аналитика;
- есть потенциал прикладной эффективности;
- в дипломе уже предусмотрен экономический раздел.

Что особенно подходит специальности:

- обоснование выбора дешёвого/бесплатного стека;
- использование цифровой обработки данных;
- практическая автоматизация анализа;
- оценка эффективности и полезности системы;
- связь культурного объекта и цифровой экономики.

Слабое место именно для этой специальности:

- экономическая часть в текущем проекте должна быть не декоративной, а прикладной;
- нужно показать, какую реальную пользу даёт система относительно ручного анализа;
- желательно сравнить затраты времени и трудоёмкость “до/после”.

Итог:

для специальности “программист в экономике” проект выглядит достаточно уместным, если делать акцент не на “чистой филологии”, а на:

- проектировании системы;
- обработке и структурировании данных;
- аналитике;
- цифровизации культурного ресурса;
- практической и экономической целесообразности.

## 13. Рекомендации по дальнейшей работе

### Для кода

- добавить тесты хотя бы на загрузку данных и метаданные;
- убрать или явно пометить `main.py` как legacy;
- ввести словарь/справочник сущностей;
- нормализовать имена и типы;
- добавить кэширование через `st.cache_data`.

### Для данных

- расширить корпус;
- исправить опечатки;
- развести персонажей, события, места и предметы;
- добавить поле источника/эпизода;
- добавить уникальные идентификаторы.

### Для диплома

- делать акцент на цифровой модели и аналитическом инструменте;
- аккуратно формулировать выводы по метрикам;
- не заявлять лишнюю полноту корпуса;
- использовать проект как пример прикладной ИТ-разработки для гуманитарных данных.

## 14. Executive Summary

Это компактный, хорошо организованный Streamlit-проект, предназначенный для построения и анализа графа знаний по `Шоҳнома`. Его главные достоинства — ясная идея, рабочий интерфейс, модульная структура и хорошая презентационная пригодность. Его основные ограничения — маленький датасет, упрощённая онтология, отсутствие тестов и семантическая хрупкость модели данных.

С технической точки зрения проект представляет собой удачный учебный/дипломный прототип аналитической системы. С точки зрения диплома по специальности `программист в экономике`, он выглядит убедительно, если позиционировать его как информационно-аналитический инструмент цифровизации культурного наследия, а не как исчерпывающее литературоведческое исследование.

Главный практический вывод: проект уже пригоден как основа дипломной работы, но для сильной защиты требует более аккуратной работы с данными, более строгих формулировок в дипломном тексте и лучшего разведения между исследовательским инструментом и научным выводом.
